# UI 草图对照审计

审计日期：2026-07-17
目标基准：`res://assets/ui/references/` 下 6 张草图
运行基准：Godot 4.6.2，Windows/OpenGL Compatibility，1920×1080

## 1. 结论摘要

当前 UI 已经具备目标草图的大部分美术词汇：深绿牌桌、木框、红绒布、金色包边、筹码、扑克牌、小丑牌和完整的按钮状态。首页与目标最接近，开局页的模块也基本齐全。核心差距集中在统一牌桌内的四个阶段页：当前把盲注、商店和结算组织成牌桌上的偏小面板，战斗区也使用了偏小的卡牌与货架，因而出现大面积空桌面、信息层级变平、标题不够强、商品与卡牌难以成为视觉主角的问题。

本轮建议不是推翻现有业务场景，而是：保留路由、状态刷新、购买/结算/出牌逻辑和既有交互状态；重做共享牌桌的构图比例、左侧状态栏的语义模式、阶段标题、主内容容器和卡牌尺度；再把已经存在但视觉职责不清的子树收敛为可复用组件。

匹配度概览（5 为最接近草图）：

| 页面 | 构图 | 层级 | 组件语言 | 综合 | 首要问题 |
|---|---:|---:|---:|---:|---|
| 首页 | 4 | 4 | 4 | 4.0 | 标题字效、缺少次要信息和装饰收口 |
| 开局页 | 3 | 3 | 3 | 3.0 | 中央模块过高且背景残留首页 HUD，信息布局松散 |
| 盲注页 | 2 | 2 | 2 | 2.0 | 三张盲注卡被压成顶部弹层，主舞台大面积空置 |
| 战斗页 | 2 | 2 | 3 | 2.3 | 手牌、小丑、消耗牌尺度过小，中央战斗叙事缺失 |
| 商店页 | 2 | 2 | 2 | 2.0 | 商品过小、货架边界弱、标题与行动层级不足 |
| 结算页 | 2 | 2 | 3 | 2.3 | 缺少庆祝焦点与底部资金汇总，内容只占上半屏 |

### 2.2 Theme

全局 Theme 为 `res://assets/ui/theme/game_theme.tres`，默认字体为 `ChillHuoGothic_F_ConBold.otf`，默认字号 24。已有 `PrimaryRedButton`、`PrimaryGoldButton`、`SecondaryRedButton`、`SecondaryGoldButton`、`SmallRedButton`、`SmallGoldButton`、`TabButton`、`IconButton`、`SurfacePanel`、`CardPanel`、`PopupSurface` 等 variation，并且按钮 Normal/Hover/Pressed/Disabled/Focus 状态完整。

Theme 的工程基础是可保留项。问题不在“没有按钮样式”，而在页面对语义样式的使用不一致：相同重要级别的行动在不同页面使用了不同尺寸、色彩和容器；盲注、商店标题仍偏普通文字；部分阶段面板退回弱化的扁平暗色表面，与草图的牌匾和货架语言断开。

## 3. 跨页面差异

### 3.1 整体构图

草图采用两类明确构图：

1. 首页/开局页是独立全屏海报式页面，不携带游戏内左侧 HUD。
2. 盲注/战斗/商店/结算共用约四分之一屏宽的左侧状态栏，右侧为几乎铺满剩余空间的阶段主舞台。

当前首页符合第一类。开局页却继续露出首页左上/右上的 HUD 和光束背景，视觉上像覆盖在首页上的设置层。游戏内四阶段虽然共享左侧 HUD，但盲注、商店、结算进一步套进中央暗色面板，面板只占右侧可用空间的一部分，使“阶段页面”看起来像“临时弹窗”。

### 3.2 视觉层级

草图的稳定层级为：页面标题牌匾 → 当前阶段/关键目标 → 卡牌或商品主体 → 主行动按钮 → 次要统计。当前层级常变成：永久木框/HUD → 大片暗绿空桌面 → 小型卡片/商品 → 普通文字标题。装饰外框比内容更抢眼，主内容本身偏弱。

### 3.3 左侧状态栏

目标并不是六页都使用左栏；左栏只属于游戏内四阶段。四张游戏内草图共享相同宽度、金色边框、分段指标和底部“比赛信息/选项”，但顶部摘要随阶段变化：盲注选择显示当前盲注和回合资源，战斗显示目标/当前分数，商店强调当前状态与资金，结算显示本局盲注和最终回合分数。

当前 `GameHudPanel` 在工程上已经统一，但视觉和语义仍是“一套战斗 HUD 到处复用”：顶部标题、回合、目标/当前、蓝红计分块和资源格几乎固定。建议保留单一组件与刷新入口，改为统一外壳 + `blind_select/battle/shop/settlement` 四种内容模式，而不是复制四套左栏。

### 3.4 标题牌匾

目标所有阶段都有清晰的红底金边主标题：盲注是宽幅“选择你的下一个盲注”，商店是巨型“商店”，结算是“回合结算”，战斗把当前盲注标题放入左栏，并用顶部小丑货架承担右侧首层标题功能。

当前首页标题已有专属底板，但文字为扁平橙色；开局页用顶部页签替代页面标题，基本合理；盲注和商店使用普通金色 Label；结算有牌匾但尺寸偏小。应抽取统一 `OrnateTitleBanner`，支持 `page/section/compact` 三个尺寸，而不是让所有页面完全同宽。

### 3.5 按钮体系

现有 Theme 的交互状态值得保留。草图的语义颜色较明确：

- 红：主流程推进、危险或离开当前阶段。
- 蓝：当前可选择的挑战/开始等强行动。
- 绿：刷新、购买、辅助正向行动。
- 金：排序、设置等工具行动。
- 灰黑：锁定或禁用。

当前首页四色按钮接近草图，战斗按钮也已具备红/金分组；但开局的开始按钮为蓝、返回为红，盲注同时出现过小蓝红按钮，商店购买按钮和大行动按钮尺度落差过大。应保留颜色角色，统一 `hero/primary/secondary/compact` 尺寸和最小点击区域。

### 3.6 卡牌展示与信息密度

草图把卡牌当成主视觉：战斗手牌约占右侧宽度的 60% 以上，小丑货架中的卡面可辨识，商店商品卡足以展示插画和描述，盲注筹码是每张卡的中心。当前 Joker 槽为 96×136、商店商品约 135×195、盲注卡约 220×400；它们在 1920×1080 的实际版面中进一步受到容器压缩，导致卡面可读性不足。

当前信息不是绝对过多，而是主次尺度不对：左栏边框和永久区占据高权重，阶段内容字号和卡片较小，空白又没有承担聚焦作用。应增大主卡与关键数值、减少同层级细边框、把次要解释收进详情/悬停，而不是继续往空白处添加文字。

### 3.7 色彩与边框

当前首页的色彩、木框和金边强度接近草图。游戏内页面同时出现全屏木框、厚重 HUD、永久区域细框、阶段暗面板、卡片细框，边框层数偏多；但阶段面板内部的区隔又偏弱，形成“外壳过重、内容过轻”。目标草图通常每个视觉组只保留一层主要金边，内部以间距、暗面色差和少量分隔线组织。

建议原则：页面外框 1 层、左栏 1 层、主内容每个大区 1 层；不要同时给容器、行、卡片槽和卡片本体都画等强金边。金色高光只用于标题、当前选择、关键金额和主按钮。

## 4. 逐页对比

### 4.1 首页

**构图差异**：当前左右分栏、左侧小丑与筹码堆、顶部标题、右侧四按钮均已对齐草图。当前标题整体略小、字形更扁平，左下主体稍收缩，底部横幅为空，右下吉祥物铭牌为空，左上货币只有图标没有数值，语言按钮缺少右侧呼应图标。

**应保留**：背景桌布、木框、卡牌/筹码主插画、按钮位置和四色语义、顶部两组 HUD、现有按钮状态反馈。

**应重做/补齐**：标题字效与比例；底部品牌副标题；货币数值；吉祥物短文案；语言按钮的双端图标；局部星光和高光强弱。

**组件化**：首页专用 `HomeHero` 和 `HomeMenuStack` 即可，不应把首页 HUD 强行复用到开局页。

### 4.2 开局页

**构图差异**：目标是独立页面，中央约六成宽的牌组选择面板，上方三页签、下方难度和两个大按钮。当前模块齐全，但主面板更宽更高，顶部贴近木框；牌组为红色而草图示例为绿色（数据差异可接受）；背景继续显示首页的皇冠、统计按钮和光束，干扰当前任务；难度区域与主面板之间空白偏大，箭头在牌组与难度两处重复出现，右侧统计使用冒号列，视觉松散。

**层级差异**：目标先读牌背和“绿色牌组”，再读四项核心属性和赌注；当前牌组名/描述/五项数据的字号接近，关键差异不突出。

**应保留**：三页签逻辑、牌组切换、牌背堆叠、属性数据、难度切换、开始/返回按钮及其状态。

**应重做**：独立背景壳；中央面板高度与上下留白；统计行对齐；难度模块与牌组模块的组合关系；只保留一组明确的牌组切换箭头。

**组件化**：`DeckCarousel`、`DeckStatsPanel`、`StakeSelector`、`TopTabs`。

### 4.3 盲注页

**构图差异**：目标标题横跨右侧顶部，三张高大的盲注卡从标题下延伸至底部行动区，右侧牌堆独立占位。当前三卡被包进上半部暗色弹层，宽度和高度都不足，右侧下半屏基本为空；底下的小丑、消耗牌和牌堆仍透出，导致“选择盲注”与“牌桌常驻物”争夺注意力。

**层级差异**：目标通过“可选”斜角、发光边框、筹码尺寸和大按钮明确当前选择；当前使用小 Badge、细描边和两枚紧凑按钮，当前卡与后续卡的差异较弱。

**应保留**：`BlindSelectPanel` 的阶段刷新逻辑、三张固定 `StageCardView`、选择/跳过/锁定状态、标签奖励和锁图。

**应重做**：阶段改为右侧主舞台而非小弹层；标题牌匾；三卡尺寸和间距；当前卡发光/缎带；右侧牌堆的明确安全区；选择/跳过按钮语义。

**组件化**：`OrnateTitleBanner`、`BlindCard`、`StateRibbon`、`RewardLine`。

### 4.4 战斗页

**构图差异**：目标右侧从上到下依次是大尺寸小丑货架、消耗牌、已出牌/计分焦点、扇形手牌和底部行动。当前小丑与消耗牌都缩在顶部细框内，手牌贴近底部且尺寸小，已出牌区域在默认状态几乎不可感知，中央大面积空桌面没有形成“等待出牌”的焦点。

**卡牌区域**：目标 8 张手牌有明显扇形、重叠和弧线，单张花色与点数在一眼距离可读；当前卡牌等高平铺、重叠较少，虽然可点击，但更像调试牌列。小丑卡也从目标的插画主视觉缩成顶部小缩略图。

**应保留**：`BattleContent`、`CardFanArea`、选择状态、排序状态、出牌/弃牌禁用逻辑、计分动画、共享 `JokerShelf/ConsumableTray/DeckArea`、详情弹窗。

**应重做**：永久区比例；卡牌最小尺寸；手牌扇形参数；已出牌与计分预览；小丑/消耗牌货架皮肤；底部按钮与手牌的距离；左栏数值权重。

**组件化**：`CardShelf`、`CardFanStage`、`ScoreEquation`、`BattleActionBar`。不应把游戏规则或动画从现有脚本复制到视觉组件。

### 4.5 商店页

**构图差异**：目标使用巨型“商店”牌匾和两个大行动按钮作为左上主入口，右侧/下方三个明确货架分别承载可购买卡牌、特殊优惠和卡牌包。当前是一个规则暗矩形，标题、按钮、商品和分区标题都偏小；商品集中在面板中央，四周空白较多，右侧牌堆与消耗牌仍透出，像在牌桌上打开的管理面板。

**信息密度**：目标每个商品面积更大、价格牌独立、货架边界清晰；当前 `ShopOfferCard` 数据完整但卡面、名称、类型、购买按钮被压在 135×195 左右，多个信息层争抢有限空间。

**应保留**：商品生成、购买、资金不足、槽位满、已售出、刷新费用、下一盲注、补充包 Overlay、共享商品卡请求信号。

**应重做**：标题与行动区；三个货架的全屏网格；商品卡尺寸和价格牌；背景帘布/筹码装饰；隐藏或弱化与商店无关的永久牌桌物件。

**组件化**：`ShopHeroHeader`、`OfferShelf`、`OfferCard`、`PriceBadge`、`ShopActionBar`。

### 4.6 结算页

**构图差异**：目标右侧主舞台从上到下完整铺开：标题、盲注摘要、左侧结算明细、右侧庆祝/提现焦点、底部资金变化汇总、继续按钮。当前大框主要占右侧上半屏，明细和提现框内缩，底部资金信息只是几行文字，右下大片空白；庆祝焦点缺少放射光、彩纸和币堆，`+$15` 的视觉权重仍不够。

**层级差异**：目标第一眼是“通过”和“提现金额”，第二眼才是逐条明细；当前先看到华丽外框和左侧文字表，成功反馈不够强。

**应保留**：结算数据结构、动态奖励行、一次性领取保护、资金前后值、继续流程。

**应重做**：全高构图；标题牌匾；左右双栏比例；庆祝视觉；底部四段式资金汇总；主按钮位置和尺寸；阶段化左栏摘要。

**组件化**：`SettlementBreakdown`、`CashoutHero`、`MoneyTransitionBar`、`RewardRow`。

## 5. 保留、重做与复用总表

| 范围 | 保留 | 重做 | 复用策略 |
|---|---|---|---|
| 路由/阶段 | `GameTableScreen` 阶段切换与刷新入口 | 阶段显示占比与遮罩策略 | 一个牌桌壳，四个阶段布局模式 |
| 左栏 | `GameHudPanel` 数据绑定 | 内部信息架构、阶段模式、边框层级 | 单组件四模式，不复制场景 |
| 标题 | 首页/结算已有资产 | 盲注、商店普通 Label 与尺寸不一致 | `OrnateTitleBanner` 三尺寸 |
| 按钮 | Theme 状态、反馈脚本、专属纹理 | 语义映射和尺寸等级 | Theme variation 为主，场景只负责组合 |
| 卡牌 | 四类卡牌脚本和数据绑定 | 最小尺寸、货架、扇形与留白 | 卡片本体与承载布局分离 |
| 商店 | 全部购买/补充包流程 | 货架构图和商品视觉 | `OfferShelf + ShopOfferCard` |
| 结算 | 金额计算、领取保护 | 明细/庆祝/汇总布局 | 数据行与视觉焦点分离 |
| 首页 | 主插画、按钮位置、背景 | 字效与次要信息 | 首页保持专用，不泛化为牌桌壳 |

## 6. 审计判定

- 当前不是“六页风格完全不同”，而是首页美术完成度较高、开局页部分继承、游戏内四阶段的主内容被统一牌桌结构压小。
- 左侧状态栏在工程结构上已统一，在目标语义和视觉层级上尚未统一；应改为统一外壳的阶段模式。
- 标题牌匾与按钮资产都已有较好基础，但页面应用规则没有形成完整设计系统。
- 优先级应先解决共享牌桌比例、左栏、标题和卡牌尺寸，再逐页换皮；先做局部装饰不会消除主要差距。
- 本轮未修改任何代码、场景、Theme 或 UI 资源；仅生成审计截图并新增规划文档。
