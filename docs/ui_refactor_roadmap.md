# UI 重构优先级与分阶段路线图

## 1. 优先级判断

### P0：先修系统性构图

- 游戏内右侧舞台占比不足，盲注/商店/结算像小弹层。
- 左侧 HUD 虽复用同一场景，但没有清晰的阶段语义模式。
- 小丑、消耗牌、手牌和商品卡尺度过小。
- 盲注/商店/结算缺少统一的主标题牌匾。

这些问题跨越四页，一旦共享壳修正，收益最大。若先做单页装饰，后续调整舞台比例时会返工。

### P1：再修页面主叙事

- 战斗：已出牌、计分、手牌、行动条的上下关系。
- 盲注：三卡状态与选择层级。
- 商店：标题/行动与三类货架。
- 结算：明细/庆祝/资金变化。

### P2：最后做完成度与响应式

- 首页标题字效、底部副标题和元信息补齐。
- 开局页独立背景、统计对齐和空白收口。
- 高光、星光、彩纸、筹码微动效。
- 1280×720、16:10、超宽屏和键盘焦点验收。

## 2. 阶段 0：建立不可回退的视觉基准

**目标**：把本次草图审计变成后续改造的验收合同。

**工作项**：

- 固定 6 个默认状态和关键交互状态的截图数据夹具。
- 修正/统一截图入口：正式验收只使用实例化后显式 `set_phase()` 的流程。
- 将设计 token、页面安全区、卡片目标尺寸和按钮语义写入验收表。
- 为每页记录 1920×1080 主基准，同时保留 1280×720 与 2520×1080 响应式基准。

**完成标准**：

- 6 页可用一条命令稳定生成不同的实际阶段画面。
- 截图中无开发占位、错误阶段或空纹理。
- 任何后续改动都能与本轮截图逐页对照。

## 3. 阶段 1：设计系统基础

**目标**：先建立全局可复用的视觉语言，不改玩法。

**工作项**：

- 在 Theme 中补齐语义颜色、字号、间距和 `hero/primary/secondary/compact` 尺寸等级。
- 新增 `OrnateTitleBanner` 的 hero/page/section/compact 变体。
- 新增 `SectionFrame`、`MetricRow`、`MetricTile`、`PriceBadge` 公共样式。
- 整理按钮语义映射：红/蓝/绿/金/禁用，保留现有完整状态资源。
- 明确纹理九宫格与程序化 StyleBox 的边界，避免重复描边。

**完成标准**：

- 同等级按钮在六页拥有一致高度、字号、边距和状态反馈。
- 标题不再由普通 Label 临时充当。
- 公共表面在组件 Gallery 中可独立检查。

**风险控制**：按 variation 小批次替换，禁止一次性重命名全部 Theme 类型；每批保留兼容别名，避免生产场景回退到默认 Button。

## 4. 阶段 2：共享牌桌壳与左侧状态栏

**目标**：消除四个游戏内页面共同的构图根因。

**工作项**：

- 保留 `GameTableScreen` 编排职责，重新定义左栏、右侧舞台和牌堆安全区。
- 把 `GameHudPanel` 原位演进为统一 `StatusSidebar`。
- 实现 `blind_select/battle/shop/settlement` 四种展示模式和显隐规则。
- 将阶段内容从“小型 BottomSheet 观感”调整为右侧全尺寸舞台；`BottomSheetHost` 只服务真正的模态内容。
- 为 Joker/Consumable/Deck 建立按阶段的显隐与弱化矩阵。

**完成标准**：

- 四阶段左栏宽度、外框、资源区和底部操作完全稳定。
- 切换阶段时主舞台不跳宽、不残留不相关卡牌货架。
- 右侧内容在 1920×1080 使用绝大多数可用面积，且不遮挡牌堆安全区。

**建议显隐矩阵**：

| 阶段 | Joker | Consumable | Deck | Phase Tint |
|---|---|---|---|---|
| 盲注 | 隐藏/弱化 | 隐藏 | 显示 | 低 |
| 战斗 | 显示 | 显示 | 显示 | 无 |
| 商店 | 可出售区按设计显示 | 隐藏/弱化 | 弱化 | 低 |
| 结算 | 隐藏/强弱化 | 隐藏 | 弱化 | 中 |

## 5. 阶段 3：战斗页优先改造

**优先原因**：战斗是停留时间最长、卡牌可读性和操作准确性最重要的页面，也是 Joker/Consumable/Deck/按钮体系的综合验收页。

**工作项**：

- 放大 Joker 与 Consumable 卡面及货架标题。
- 扩大手牌承载区，调整 `CardFanArea` 的卡片尺寸、重叠、弧线和选中抬升。
- 建立可感知的已出牌/牌型/计分舞台；默认态低对比，出牌态高对比。
- 让 `ScoreEquation` 与左栏当前分形成明确主次，不重复展示同等级数字。
- 重排底部行动条，统一主行动、危险行动和工具行动尺寸。

**保留不动**：牌型判断、计分公式、卡牌选择集合、排序规则、出牌/弃牌次数、动画触发顺序。

**完成标准**：

- 8 张手牌在 1280×720 仍可辨识点数与花色，在 1920×1080 达到目标主视觉尺度。
- 默认、单选、多选、排序、出牌禁用、弃牌禁用截图全部无跳位。
- 中央空白是有意的出牌舞台，不再像遗漏内容。

## 6. 阶段 4：盲注页改造

**工作项**：

- 将标题提升为 page 牌匾并占据右侧顶部中心。
- 把三张 `StageCardView` 放大为主舞台卡，统一目标/奖励/标签行。
- 建立 Current/Next/Locked 的持续状态：发光、缎带、锁图和按钮组合。
- 调整牌堆与底部装饰，确保右侧信息不悬空。

**保留不动**：盲注数据、选择/跳过规则、Boss 解锁、标签奖励。

**完成标准**：三卡在不读文字时也能仅凭颜色、发光和锁定图判断状态；行动按钮不低于最小点击尺寸。

## 7. 阶段 5：商店页改造

**工作项**：

- 建立巨型商店标题与“下一盲注/刷新”行动组。
- 用三个 `OfferShelf` 明确分组：卡牌、优惠券、补充包。
- 放大 `ShopOfferCard`，让插画、名称、价格成为第一层；类型和长描述降为第二层。
- 价格独立为 `PriceBadge`，资金不足、槽位满、售出必须可区分。
- Pack Overlay 保持真正模态，验证打开/选择/跳过流程。

**保留不动**：商店生成、价格、刷新递增、购买条件、出售、补充包和优惠券业务。

**完成标准**：默认、Hover、资金不足、槽位满、售出、补充包打开状态均能从截图直接辨识；商品不与右侧永久牌堆互相遮挡。

## 8. 阶段 6：结算页改造

**工作项**：

- 用全高舞台重排标题、盲注摘要、明细、提现焦点、资金变化和继续按钮。
- `SettlementBreakdown` 用数据驱动 `RewardRow` 消除隐藏行占位风险。
- `CashoutHero` 增加金币、高光和彩纸动效，但保持金额可读。
- `MoneyTransitionBar` 固定显示基础收入、奖励合计、领取前和领取后资金。
- 保持领取按钮防重复与状态稳定。

**保留不动**：结算金额来源、利息/标签/优惠券规则、一次性领取保护、阶段跳转。

**完成标准**：第一眼明确“通过 + 总收入”，第二眼能追溯收入来源；领取前后无横向跳字，禁用态可读。

## 9. 阶段 7：首页与开局页收口

### 首页

- 放大并提升标题字效，恢复草图的金色立体感。
- 补齐货币数值、底部副标题、吉祥物铭牌和语言按钮右侧图标。
- 调整星光和筹码高光，不改变已接近目标的左右构图。

### 开局页

- 切换为独立桌面背景，不再透出首页 HUD。
- 收紧中央面板高度、统计行和难度区间距。
- 明确牌组切换与难度切换两组箭头的空间归属。
- 保留页签、继续禁用、挑战与开始 Hover 状态。

**完成标准**：两页视觉上都是独立正式页面；首页无空铭牌，开局页无首页残留控件，1280×720 的开始/返回不重叠。

## 10. 阶段 8：响应式、可访问性与验收

**分辨率矩阵**：1280×720、1600×900、1920×1080、2560×1440、1920×1200、2520×1080。

**状态矩阵**：

- 首页：开始/设置/语言 Hover。
- 开局：继续禁用/启用、挑战、切换牌组、开始 Hover。
- 战斗：默认、单选、多选、两种排序、出牌/弃牌禁用。
- 盲注：可选、跳过、锁定、Boss。
- 商店：默认、资金不足、槽位满、售出、补充包打开。
- 结算：完成、领取后禁用。

**检查项**：

- 键盘 Focus 与 Hover 可区分。
- 所有按钮点击区域至少 56×56。
- 关键文字和状态不只依赖颜色。
- 长中文不溢出，数值增长不跳布局。
- 超宽屏不拉伸卡面，16:10 不切掉底部行动。

## 11. 交付切片与回滚边界

建议每个阶段独立提交：

1. Tokens/Theme/公共组件。
2. 共享牌桌壳与 StatusSidebar。
3. Battle。
4. Blind Select。
5. Shop。
6. Settlement。
7. Home/Run Setup。
8. Responsive/Polish。

每个提交只改对应场景、Theme/StyleBox、必要资产和测试截图；业务规则脚本若无必要不动。这样任一页面可单独回滚，不影响已经稳定的其他阶段。

本轮仅完成读取、实际运行截图、差异分析、设计系统提炼、组件方案和路线图。未实施任何代码、场景、Theme 或资源改造。
